## 1. 사용한 도구
| 도구 | 버전/플랜 | 역할 |
|------|-----------|------|
| **Make** (구 Integromat) | Free 플랜 (1,000 ops/월) | 노드형 시각 자동화 |
| **Zapier** | Free 플랜 (100 tasks/월) | 리스트형 자동화 |
| Google Forms / Sheets | 무료 | 트리거·데이터 저장 |
| Discord (네이티브 모듈) | 무료 | 알림 채널 (Send a Message to a Channel) |
| Google Gemini (2.5 Flash-Lite) | 무료 API 키 | (보너스) AI 맞춤 메시지 생성 |

## 2. 동일하게 구현한 워크플로우 구조
- **Trigger**: Google Forms 새 응답 감지
- **AI Action(보너스)**: Google Gemini로 신청자 맞춤 안내 메시지 생성
- **조건 분기**: 사전 퀴즈 점수 80점 기준
- **Action 1**: Google Sheets 행 추가 (승인자 / 대기자 시트)
- **Action 2**: Discord 알림 발송 (승인 / 보완 안내)

> 두 도구 모두 **동일한 입력(폼 응답) → 동일한 판정(80점) → 동일한 출력(시트+알림)** 구조.
> 차이는 "분기를 표현하는 방식"과 "무료 플랜 제약"에서 발생함.

> 📌 **구현 화면(워크플로우 구성·실행 결과 스크린샷)은 [프로젝트 2 문서](05_프로젝트2_자유주제.md)로 이전**하였음. 본 보고서는 **두 도구의 비교 분석**에 집중함.

## 3. 비교 항목 (9개 항목)

| # | 비교 항목 | Make | Zapier |
|---|-----------|------|--------|
| 1 | **UI/UX** | 노드/캔버스 기반. 전체 흐름을 그림처럼 한눈에 파악 | 위→아래 리스트(단계) 기반. 순차적이라 직관적 |
| 2 | **설정 난이도** | 데이터 매핑·Router 개념 등 초기 학습곡선 있음 | 마법사식 안내로 입문자가 빠르게 완성 |
| 3 | **조건 분기 방식** | **Router** 무료 지원 → 한 시나리오에 다중 경로 | **Paths(라우터) 유료** → 무료는 Filter 2개/Zap 2개로 우회 |
| 4 | **연동 서비스 범위** | 2,000+ 앱, HTTP/JSON 등 개발자 친화 모듈 강함 | 7,000+ 앱, 대중 SaaS 커버리지 넓음 |
| 5 | **무료 플랜 범위** | 1,000 ops/월, 시나리오 무제한, Router·다단계 허용 | 100 tasks/월, 2단계 Zap, Webhooks·Paths 유료 |
| 6 | **실행 로그 확인** | History에서 모듈별 입력/출력 버블(데이터)까지 상세 확인 | Zap History에서 단계별 성공/실패·데이터 확인 |
| 7 | **재시도·에러 처리** | Error handler / 롤백·재시도 경로를 시각적으로 설계 가능 | Autoreplay(재시도)는 유료, 무료는 오류 이메일 통지 수준 |
| 8 | **AI(Gemini) 연동** | Gemini 모듈에 Messages·Model 세부 지정, 출력 `Result` 재사용 용이 | AI 액션이 단계로 삽입돼 직관적, 세부 파라미터는 상대적으로 단순 |
| 9 | **데이터 매핑 편의** | 중첩(`answers[1].value`) 등 세밀 제어 가능하나 손이 감 | 필드가 평면적으로 노출돼 초보자가 고르기 쉬움 |

## 4. 각 도구의 장단점

### Make
**장점**
- 복잡한 분기·반복·에러처리를 **시각적으로** 설계 가능(Router/Iterator/Error handler 무료).
- 실행 로그가 **모듈 단위 데이터**까지 보여 디버깅에 유리.
- ops 기준 무료 범위가 상대적으로 넉넉(1,000/월).

**단점**
- 데이터 매핑·연산자 개념 등 **초기 학습곡선**이 있음.
- 화면이 넓어질수록 대형 시나리오는 한눈에 보기 어려워짐.

### Zapier
**장점**
- **입문 난이도 최저**. 단계별 마법사로 5분 내 첫 자동화 완성.
- 연동 앱 수가 가장 많아 **대중 SaaS 커버리지** 우수.
- 문서·템플릿·커뮤니티가 풍부.

**단점**
- **핵심 분기 기능(Paths)·Webhooks·재시도(Autoreplay)가 유료.**
- 무료 tasks(100/월)가 빠르게 소진되고, 분기 구현 시 Zap 수가 늘어 tasks 소비 증가.

## 5. 어떤 상황에 적합한가 (의견)

| 상황 | 추천 | 이유 |
|------|------|------|
| 복잡한 분기·에러처리·데이터 가공이 많은 업무 | **Make** | Router/Error handler가 무료이고 로그가 상세 |
| 개발지식 적은 담당자가 빠르게 표준 연동만 필요 | **Zapier** | 마법사 UI로 학습 없이 즉시 구축 |
| 실행량(트리거 빈도)이 많고 예산 최소화 | **Make** | 동일 작업당 무료 ops 여유가 큼 |
| 마이너/대중 SaaS 연동 커넥터가 관건 | **Zapier** | 앱 커버리지 7,000+로 최다 |
| 팀 협업·비개발 부서 도입 용이성 | **Zapier** | 직관적 UI로 온보딩 쉬움 |

**종합 의견**: 이번 워크플로우처럼 **조건 분기와 에러 처리를 무료로 온전히 구현**하려면 Make가 유리했다. Zapier는 구축 자체는 가장 쉬웠지만, 무료 플랜에서 분기(Paths)와 재시도가 막혀 **Zap을 2개로 쪼개는 우회**가 필요했다. 따라서 "학습 편의 우선 → Zapier", "기능·비용 효율 우선 → Make"로 선택하는 것이 합리적이다.

## 6. 유료 기능 사용 여부 및 무료 대안
- 본 과제는 **전부 무료 플랜으로 완수**함. (Make Free, Zapier Free, Google Gemini 무료 API, Discord 무료)
- Zapier의 Paths(유료)는 **Filter 2개 + Zap 2개**라는 무료 대안으로 대체.
- Zapier Webhooks(유료)는 **Google Forms 네이티브 트리거**로 회피.
- 재시도(Autoreplay, 유료)는 **오류 발생 시 알림 통지 + 임시 시트 적재**(Make Error handler)로 대체.

### 구현 중 겪은 실제 문제와 해결 (공통 학습)
| 문제 | 원인 | 해결 |
|------|------|------|
| Router/Filter 두 갈래 모두 통과 실패(`⊘0`) | Google Forms 응답이 `answers[].value`로 깊게 중첩 → 얕은 매핑은 빈 값 | `answers[1].value`까지 파고들어 매핑, 연산자를 Numeric으로 지정 |
| Gemini 503(과부하) | 모델 일시 과부하 | 재시도 / 모델 변경 |
| Gemini 429(quota 초과, `limit:0`) | `-lite` 프리뷰 모델의 무료 일일 할당이 0 | **`Gemini 2.0 Flash` 등 무료 할당 있는 모델로 변경** |

> 교훈: 생성형 AI를 Action으로 붙일 땐 **무료 모델의 rate limit·일일 quota**를 반드시 확인해야 하며, AI는 보너스 요소이므로 **실패 시 기본 문구로 대체하는 에러 경로**를 함께 설계하는 것이 안전하다.
